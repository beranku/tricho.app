# Architecture

This is the abridged operator-facing version of the
[`marketing-site-and-pwa-split`](./openspec/changes/marketing-site-and-pwa-split/)
OpenSpec change. The canonical detail (rationale, alternatives, threat
model, every requirement) lives there. This file is the 2-page summary.

## One origin, two surfaces

`tricho.app` serves two different things from one Cloudflare Pages
project (`tricho`):

| URL              | Source              | What it is                                    |
|------------------|---------------------|-----------------------------------------------|
| `/`              | `web/`              | Marketing landing                             |
| `/about`         | `web/`              | About page                                    |
| `/pricing`       | `web/`              | Pricing tiers                                 |
| `/blog/**`       | `web/` (MDX)        | Blog                                          |
| `/help/**`       | `web/` (MDX)        | Help portal                                   |
| `/legal/**`      | `web/`              | Privacy / Terms / GDPR                        |
| `/sitemap.xml`   | `@astrojs/sitemap`  | SEO surface (excludes `/app/*`)               |
| `/robots.txt`    | `web/public/`       | Disallows `/app/`                             |
| `/blog/rss.xml`  | `@astrojs/rss`      | RSS feed                                      |
| `/manifest.webmanifest` | `shared/`    | Web App Manifest, `start_url: '/app/'`        |
| `/sw.js`         | `shared/`           | Thin pass-through SW (scope `/`)              |
| `/icons/*`       | `shared/`           | App + favicon icons                           |
| `/og/*`          | `shared/` + per-post | Open Graph images                            |
| `/app/`          | `app/`              | PWA shell (Astro + React + vite-pwa)          |
| `/app/sw.js`     | `app/` (Workbox)    | Full PWA SW (scope `/app/`)                   |

## Two service workers, two scopes, no overlap

- **Thin SW** at `/sw.js` (scope `/`): 15 lines, three handlers, **no
  caching**. Sole purpose: satisfy install criteria so Chrome / Edge
  surface the install prompt on the marketing landing.
- **Full SW** at `/app/sw.js` (scope `/app/`): generated by
  `@vite-pwa/astro` with `registerType: 'prompt'`,
  `clientsClaim: false`, `skipWaiting: false`. Owns offline cache,
  asset versioning, and the user-controlled update prompt
  (the `<UpdateBanner>` in the unlocked shell).

Updates are **never silent**. When a new SW finishes installing it sits
waiting; the user sees "Nová verze připravena — restartovat" and chooses
when to reload. This avoids racing with the in-memory DEK.

## Two packages, one merge

Each top-level package builds independently. There is **no monorepo
tooling** (Turborepo, Nx, npm/pnpm workspaces, Lerna).

```
web/  npm install && npm run build  →  web/dist/   (static marketing)
app/  npm install && npm run build  →  app/dist/   (PWA, base: '/app/')
```

A single Node script combines them:

```
scripts/merge-dist.mjs:
  dist/                    ← web/dist/**
  dist/app/                ← app/dist/**
  dist/manifest.webmanifest ← shared/manifest.webmanifest
  dist/sw.js               ← shared/sw.js
  dist/icons/              ← shared/icons/**
  dist/og/                 ← shared/og/**
  dist/_headers            ← _headers
  dist/_redirects          ← _redirects
```

`scripts/validate-build.mjs` then asserts: marketing index links the
manifest, registers `/sw.js`; both SWs exist at the right paths;
manifest's `start_url` and `scope` are `/app/`; sitemap, redirects,
headers all present. Exits non-zero on any failure.

## Cloudflare Pages routing

`_redirects`:

```
/app    /app/            301
/app/*  /app/index.html  200
```

The `200` rewrite gives the SPA fallback for `/app/*` deep links and
reloads. The `301` normalizes the bare `/app` form so it agrees with the
manifest's trailing-slash `start_url`.

`_headers` (highlights):

- Both SWs: `no-cache, no-store, must-revalidate` + `Service-Worker-Allowed`.
- Manifest: 1-hour cache + `Content-Type: application/manifest+json`.
- Hashed assets (`/_astro/*`, `/app/_astro/*`, `/app/assets/*`, `/icons/*`,
  `/og/*`): `max-age=31536000, immutable`.
- HTML: `max-age=0, must-revalidate`.
- Global: `X-Content-Type-Options`, `X-Frame-Options: DENY`,
  `Referrer-Policy`, `Permissions-Policy`, HSTS.
- `/app/*`: tight CSP scoped to the app shell (the surface that holds
  plaintext in memory). Marketing pages stay under the global headers.

## CI/CD

`.github/workflows/ci.yml`:

1. `changes` job (uses `dorny/paths-filter@v3`) detects which side changed.
2. `test-web` and `test-app` run independently and in parallel; either
   side is skipped if its filter didn't match.
3. `build-and-deploy` downloads (or rebuilds) both artifacts, runs the
   merge + validate scripts, and deploys via
   `cloudflare/wrangler-action@v3 pages deploy dist --project-name=tricho`.
4. PR runs deploy to `<branch>.tricho.pages.dev` and post the URL as a PR
   comment. Pushes to `main` deploy to production (`tricho.app`).

`.github/workflows/tests.yml` runs the existing test pyramid (smoke,
unit, component, backend-unit, backend-integ, e2e, coverage-gate) — its
path filters and `working-directory` are rebased to `app/`.

`.github/workflows/release-app.yml` triggers on `app-v*` tag push and
publishes a GitHub Release with auto-generated notes from `app/`-scoped
commits since the previous `app-v*` tag.

## Required GitHub secrets

| Secret                     | Purpose                                          |
|----------------------------|--------------------------------------------------|
| `CLOUDFLARE_API_TOKEN`     | Pages-Edit-only token; deploys via wrangler.    |
| `CLOUDFLARE_ACCOUNT_ID`    | Cloudflare account that owns the `tricho` project. |
| `SOPS_AGE_KEY` (existing)  | Decrypts `secrets/*.sops.yaml` for the e2e stack. |
| `GITHUB_TOKEN` (auto)      | PR-comment posting + release creation.           |

## Rollback

Cloudflare Pages keeps every deployment. One click in the dashboard
(Workers & Pages → tricho → Deployments → "Rollback to this deployment")
restores any prior deploy in seconds. No GitHub Actions run is required.

## Versioning

| Tag        | Drives                                                |
|------------|-------------------------------------------------------|
| `app-v*`   | `release-app.yml` → GitHub Release for the PWA.       |
| `web-v*`   | Reserved by convention; not wired today.              |

`app/package.json` `version` is canonical for the PWA. The build embeds
`__APP_VERSION__`, `__APP_BUILD_TIME__`, `__APP_COMMIT__` via Vite's
`define`; Settings → "O aplikaci" displays them and links to the
matching GitHub Release.

## Zero-knowledge invariants — preserved

This change is **routing + build/deploy only**. The crypto envelope, the
AAD binding, the `wrappedDekRs / wrappedDekPrf / wrappedDekPin` keystore,
the `vault-state` doc shape, the `tricho-auth` JWT contract, the
`couch_peruser` model — none of those are touched. The marketing site is
plain HTML+CSS that never sees plaintext, never holds a DEK, never opens
a `VaultDb`, and never speaks to CouchDB or `tricho-auth`. The IndexedDB
origin (`tricho.app`) is unchanged, so any keystore data already
persisted survives the cutover.
