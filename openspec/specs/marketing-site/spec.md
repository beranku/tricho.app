## Requirements
### Requirement: Marketing surface is fully static, pre-rendered HTML

The marketing site (root paths `/`, `/about`, `/pricing`, `/blog/**`, `/help/**`, `/legal/**`, `/404`) MUST be produced by Astro 5 with `output: 'static'`. Every page MUST emit complete HTML with full `<head>` metadata at build time. The site MUST NOT depend on client-side JavaScript to render any user-visible text, navigation, or layout — JavaScript MAY enhance interactivity (the iOS install banner reveal, the standalone-mode "Launch app" link) but text content MUST be present in the initial HTML response. Hydrated islands MAY be used sparingly; their JavaScript bundles MUST NOT exceed 50 KB compressed in aggregate per page.

#### Scenario: Crawler can read the landing page without JavaScript
- **GIVEN** a request to `https://tricho.app/` from a client that does not execute JavaScript
- **WHEN** the response is parsed
- **THEN** the visible product description, value proposition, pricing, and CTA copy are present in the HTML
- **AND** the `<title>`, `<meta name="description">`, and `<link rel="canonical">` are present
- **AND** Open Graph and Twitter Card metadata are present

#### Scenario: Hydrated islands stay under the JavaScript budget
- **GIVEN** the production build of the landing page
- **WHEN** the total compressed size of all JavaScript shipped to the client is measured
- **THEN** it is less than 50 KB

### Requirement: Per-page SEO metadata is layout-driven, not hand-written

Every marketing page MUST render its `<head>` via a single Astro layout (`web/src/layouts/Base.astro`) which accepts `title`, `description`, `canonical?`, `ogImage?`, `ogType?`, `noIndex?` props. Pages MUST NOT hand-write `<meta>`, `<title>`, or `<link rel="canonical">` tags directly. Blog and help pages MUST derive these props from MDX frontmatter via Astro Content Collections.

#### Scenario: A new page added through the layout has correct head metadata
- **GIVEN** a new `web/src/pages/about.astro` that uses `<BaseLayout title="O nás" description="...">`
- **WHEN** the page builds
- **THEN** the emitted `<head>` contains `<title>O nás</title>`, the `<meta name="description">` matches, and `<link rel="canonical">` is the absolute URL of the page

#### Scenario: A blog post derives metadata from frontmatter
- **GIVEN** a blog MDX file with frontmatter `{ title: 'X', description: 'Y', date: '2026-04-26' }`
- **WHEN** the post is built
- **THEN** the page's `<title>` is `X`, `<meta description>` is `Y`, and the JSON-LD `Article` block contains `datePublished: '2026-04-26'`

### Requirement: Open Graph and Twitter Card metadata appear on every page

Every marketing page MUST emit:

- `<meta property="og:title">` matching the page title
- `<meta property="og:description">` matching the page description
- `<meta property="og:image">` referencing either the page-specific OG image (blog posts) or `/og/default.png` (everything else)
- `<meta property="og:url">` matching the canonical URL
- `<meta property="og:type">` (`website` for landing/about/pricing/legal, `article` for blog posts)
- `<meta property="og:site_name">` set to `Tricho`
- `<meta property="og:locale">` set to `cs_CZ`
- `<meta name="twitter:card" content="summary_large_image">`
- `<meta name="twitter:title">`, `<meta name="twitter:description">`, `<meta name="twitter:image">` matching the OG values

#### Scenario: Landing page OG renders the default OG image
- **GIVEN** the landing page builds
- **WHEN** the `<head>` is inspected
- **THEN** `<meta property="og:image" content="https://tricho.app/og/default.png">` is present
- **AND** the image dimensions are 1200×630

#### Scenario: Blog post OG renders the per-post OG image
- **GIVEN** a blog post with frontmatter `ogImage: '/og/blog/welcome-to-tricho.png'`
- **WHEN** the page builds
- **THEN** `<meta property="og:image" content="https://tricho.app/og/blog/welcome-to-tricho.png">` is present

### Requirement: Site-wide sitemap, robots.txt, and RSS are generated at build time

The marketing site MUST emit:

- `/sitemap.xml`: generated by `@astrojs/sitemap` from all pages under `web/src/pages/`, EXCLUDING any path matching `/app/**` (the PWA shell is not crawler-targeted content).
- `/robots.txt`: a static file allowing all user-agents on `/`, disallowing `/app/`, and pointing at `https://tricho.app/sitemap.xml`.
- `/blog/rss.xml`: generated by `@astrojs/rss` from the blog content collection, ordered newest-first.

#### Scenario: Sitemap excludes the app shell
- **GIVEN** the production build
- **WHEN** `dist/sitemap.xml` is parsed
- **THEN** no `<loc>` element references a URL under `/app/`

#### Scenario: robots.txt disallows /app/
- **GIVEN** the production build
- **WHEN** `dist/robots.txt` is read
- **THEN** it contains the directive `Disallow: /app/`
- **AND** it contains the directive `Sitemap: https://tricho.app/sitemap.xml`

#### Scenario: RSS feed is reachable and well-formed
- **GIVEN** at least one blog post exists in the content collection
- **WHEN** `dist/blog/rss.xml` is fetched
- **THEN** it is a valid RSS 2.0 document
- **AND** it contains an `<item>` per blog post with `<title>`, `<description>`, `<link>`, `<pubDate>`, `<guid>`

### Requirement: JSON-LD structured data is rendered conditionally per page type

The marketing site MUST emit JSON-LD `<script type="application/ld+json">` blocks per page type:

- **All pages**: an `Organization` block with `@type: 'Organization'`, `name: 'Tricho'`, `url: 'https://tricho.app'`, `logo: 'https://tricho.app/icons/icon-512.png'`.
- **`/` and `/pricing`**: also a `SoftwareApplication` block with `@type: 'SoftwareApplication'`, `applicationCategory: 'BusinessApplication'`, `operatingSystem: 'Web'`, `offers: { @type: 'Offer', price: '0' or relevant Pro price }`.
- **Blog posts**: an `Article` block with `@type: 'Article'`, `headline`, `datePublished`, `dateModified`, `author`, `image`.

Pages without a defined block type MUST NOT emit a misleading or empty JSON-LD block.

#### Scenario: Landing page emits Organization and SoftwareApplication
- **GIVEN** the landing page builds
- **WHEN** the `<head>` is parsed
- **THEN** there are exactly two `<script type="application/ld+json">` blocks
- **AND** one has `@type: 'Organization'` and the other has `@type: 'SoftwareApplication'`

#### Scenario: Blog post emits Article JSON-LD
- **GIVEN** a blog post page builds
- **WHEN** the `<head>` is parsed
- **THEN** an `Article` JSON-LD block is present with all required fields populated

### Requirement: Marketing site shares brand identity with the PWA shell

The marketing site MUST render with the same brand wordmark, paper-grain background texture, Fraunces serif headings, Patrick Hand body type for handwritten copy, Geist sans for UI text, and copper accents (`var(--copper)`, `var(--copper-mid)`, `var(--copper-border)`) as the PWA shell. The brand-color tokens MUST be defined in `web/src/styles/tokens.css` and MUST match the values in the app's tokens by hex value (a parity check is part of the merge build's validation, even if duplicated across packages).

#### Scenario: Landing page uses the same theme color as the PWA manifest
- **GIVEN** the landing page builds
- **WHEN** the `<meta name="theme-color">` value is compared to `shared/manifest.webmanifest`'s `theme_color`
- **THEN** they are identical

#### Scenario: Brand fonts load on the marketing landing
- **GIVEN** the landing page renders in a browser
- **WHEN** the computed font-family of an `<h1>` is inspected
- **THEN** it resolves to `Fraunces`
- **AND** the font weight is consistent with the PWA shell's `<h1>` styling

### Requirement: iOS-Safari install banner is conditional and dismissable

The marketing landing MUST include an `InstallBanner` component that is hidden by default and revealed only on iOS Safari when `navigator.standalone === false`. The banner MUST display Add-to-Home-Screen instructions in Czech ("Klepni na sdílení a vyber 'Přidat na plochu'"). The banner MUST be dismissable; dismissal MUST persist in `localStorage['install-banner-dismissed']` so the banner does not reappear on subsequent visits. The banner MUST NOT render on Android browsers, desktop browsers, or in-app browsers (Instagram, Facebook in-app), and MUST NOT render when the page is already in standalone mode.

#### Scenario: Banner shows on iOS Safari in browser mode
- **GIVEN** a visit to `https://tricho.app/` from iOS Safari
- **AND** `navigator.standalone === false`
- **AND** `localStorage['install-banner-dismissed']` is unset
- **WHEN** the page renders
- **THEN** the install banner is visible

#### Scenario: Banner stays hidden on Android Chrome
- **GIVEN** a visit to `https://tricho.app/` from Android Chrome
- **WHEN** the page renders
- **THEN** the install banner has the `hidden` attribute set
- **AND** it is not visually present

#### Scenario: Dismissed banner does not return on next visit
- **GIVEN** the banner is visible on iOS Safari
- **WHEN** the user clicks the dismiss button
- **THEN** `localStorage['install-banner-dismissed']` is set to `'1'`
- **AND** on the next page load the banner remains hidden

### Requirement: Marketing landing surfaces a "Launch app" affordance when running standalone

The marketing landing MUST detect `window.matchMedia('(display-mode: standalone)').matches` and, when true, render a primary "Otevřít aplikaci" link/button targeting `/app/`. When standalone mode is false, this affordance MUST NOT be present. The detection MUST happen client-side and MUST NOT cause layout shift on first paint (the affordance is hidden by default and revealed by JavaScript).

#### Scenario: Standalone visit shows Launch app
- **GIVEN** the marketing landing is opened from inside the installed PWA (display-mode standalone)
- **WHEN** the page renders
- **THEN** an "Otevřít aplikaci" affordance is visible
- **AND** clicking it navigates to `/app/`

#### Scenario: Browser-mode visit hides Launch app
- **GIVEN** the marketing landing is opened in a regular browser tab
- **WHEN** the page renders
- **THEN** no "Otevřít aplikaci" affordance is visible

### Requirement: Marketing site meets Core Web Vitals budget on the landing page

The landing page MUST hit, on a 4G mobile profile, Lighthouse scores: Performance ≥ 95, Accessibility ≥ 95, Best Practices = 100, SEO = 100. LCP MUST be < 2.0s, CLS MUST be < 0.1, TBT MUST be < 150ms, and total HTML+CSS+critical JS for `/` MUST be < 100 KB compressed.

#### Scenario: Lighthouse run on the landing page passes the budget
- **GIVEN** a deployed PR preview URL
- **WHEN** Lighthouse mobile is run against `/`
- **THEN** all four category scores meet the thresholds above
- **AND** LCP < 2000ms, CLS < 0.1, TBT < 150ms

#### Scenario: Critical resource size budget is enforced at build time
- **GIVEN** the landing page builds
- **WHEN** the size of `<head>`-blocking CSS plus inline JS plus the initial HTML is measured (compressed)
- **THEN** the total is less than 100 KB

### Requirement: 404 page renders with site chrome and a back-to-home link

A `/404.html` page MUST be present in the build output and served by Cloudflare Pages for any unmatched marketing-site route. The 404 page MUST share the marketing-site layout (header, footer, fonts, brand colors), display a friendly Czech message, and contain a link back to `/`.

#### Scenario: Unknown route renders the styled 404
- **GIVEN** a request to `https://tricho.app/this-does-not-exist`
- **WHEN** the response is parsed
- **THEN** the HTTP status is 404
- **AND** the response body contains the marketing-site header and footer
- **AND** a link to `/` is present

### Requirement: Blog and help skeletons are present, content is editorial workstream

The blog (`/blog/`, `/blog/[slug]`) and help (`/help/`, `/help/[...slug]`) routes MUST be wired and rendering. The blog MUST ship with at least one welcome post; the help MUST ship with at least one welcome help article. Both MUST use Astro Content Collections backed by MDX. Editorial content beyond the welcome posts is explicitly NOT part of this change and is tracked as a separate workstream.

#### Scenario: Blog index lists the welcome post
- **GIVEN** the production build
- **WHEN** `/blog/` is fetched
- **THEN** the page lists at least one post by title with date and excerpt
- **AND** clicking the post navigates to its slug-based detail page

#### Scenario: Help index lists the welcome article
- **GIVEN** the production build
- **WHEN** `/help/` is fetched
- **THEN** the page lists at least one help article
- **AND** clicking the article navigates to its slug-based detail page

### Requirement: Pricing page reflects the live billing-plans contract

The `/pricing` page MUST list the plan tiers defined in the existing `billing-plans` capability (Free, Pro, etc. — current tier names and prices read from `billing-plans` spec). Each tier MUST display: name, monthly and annual price (Czech crowns), included device count, included feature list, CTA. The CTA for Free MUST link to `/app/` (start using); the CTA for paid plans MUST link to `/app/` with `?intent=upgrade` so the in-app billing UI knows to surface the upgrade screen.

#### Scenario: Pricing reflects the current plan catalog
- **GIVEN** the `billing-plans` spec defines plans Free and Pro
- **WHEN** `/pricing` is rendered
- **THEN** each plan from `billing-plans` appears on the page with its current price
- **AND** the CTA for paid plans links to `/app/?intent=upgrade`

#### Scenario: Pricing page passes structured-data validation
- **GIVEN** the `/pricing` page builds
- **WHEN** the JSON-LD is extracted and validated against schema.org's `SoftwareApplication` `Offer` schema
- **THEN** validation passes with no errors
