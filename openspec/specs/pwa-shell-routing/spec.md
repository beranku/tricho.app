## Requirements
### Requirement: Single shared Web App Manifest at /manifest.webmanifest

The application MUST be installable from a single shared manifest served at `/manifest.webmanifest`. The manifest source of truth lives at `shared/manifest.webmanifest` and the merge build copies it to `dist/manifest.webmanifest`. The manifest MUST set `start_url: '/app/'`, `scope: '/app/'`, `display: 'standalone'`, `name: 'Tricho'`, `short_name: 'Tricho'`, and reference shared icons at `/icons/icon-192.png`, `/icons/icon-512.png`, and the corresponding maskable variants. The manifest's `theme_color` and `background_color` MUST match the values used by both the marketing site's `<meta name="theme-color">` and the PWA shell's loading screen, so the transition from the install splash to the first paint shows no color flash.

#### Scenario: Manifest is served at the apex
- **GIVEN** the production deploy
- **WHEN** `https://tricho.app/manifest.webmanifest` is fetched
- **THEN** the response status is 200
- **AND** the `Content-Type` header is `application/manifest+json`
- **AND** the JSON body contains `start_url: '/app/'` and `scope: '/app/'`

#### Scenario: Theme color is consistent across surfaces
- **GIVEN** the production deploy
- **WHEN** the manifest's `theme_color`, the marketing landing's `<meta name="theme-color">`, and the PWA shell's `<meta name="theme-color">` are read
- **THEN** all three values are identical

### Requirement: Both surfaces link the shared manifest

The marketing landing (`/`) and the PWA shell entry HTML (`/app/`) MUST both render `<link rel="manifest" href="/manifest.webmanifest">` in their `<head>`. Other marketing pages (`/about`, `/pricing`, `/blog/**`, `/help/**`, `/legal/**`) MAY also link the manifest but are NOT required to.

#### Scenario: Install prompt fires from the marketing landing
- **GIVEN** a Chromium-based browser visits `https://tricho.app/`
- **AND** the install criteria are met (manifest reachable, SW registered, served over HTTPS)
- **WHEN** the page loads
- **THEN** the browser surfaces the native install affordance (address-bar install icon or installability banner)

#### Scenario: Install prompt fires from the app shell
- **GIVEN** a Chromium-based browser visits `https://tricho.app/app/` in browser mode
- **AND** the install criteria are met
- **WHEN** the page loads
- **THEN** the browser surfaces the native install affordance

### Requirement: Thin pass-through service worker registered at /sw.js scope /

The system MUST register a pass-through service worker at `/sw.js` with scope `/`. The SW source lives at `shared/sw.js` and contains exactly three handlers: `install` (calls `self.skipWaiting()`), `activate` (calls `event.waitUntil(self.clients.claim())`), and `fetch` (returns nothing — does NOT call `event.respondWith`, allowing the browser to handle every request normally). The thin SW MUST NOT cache any response. The marketing landing's `<head>` MUST include the inline registration script that calls `navigator.serviceWorker.register('/sw.js', { scope: '/' })`. The thin SW MUST be served with HTTP headers `Cache-Control: no-cache, no-store, must-revalidate` and `Service-Worker-Allowed: /`.

#### Scenario: Thin SW registers with root scope
- **GIVEN** the marketing landing loads in a browser that supports service workers
- **WHEN** the `load` event fires and the registration script runs
- **THEN** `navigator.serviceWorker.getRegistration('/')` resolves to a registration whose `active.scriptURL` ends with `/sw.js` and whose `scope` is `https://tricho.app/`

#### Scenario: Thin SW does not intercept requests
- **GIVEN** the thin SW is active
- **WHEN** any fetch (HTML, CSS, JS, API) is made
- **THEN** the SW's `fetch` handler does not call `event.respondWith`
- **AND** the browser's default network handling completes the request
- **AND** no cache entry is created in any SW-managed cache

#### Scenario: Thin SW is served with no-cache headers
- **GIVEN** the production deploy
- **WHEN** `https://tricho.app/sw.js` is fetched
- **THEN** the `Cache-Control` response header is `no-cache, no-store, must-revalidate`
- **AND** the `Service-Worker-Allowed` response header is `/`

### Requirement: Full PWA service worker registered at /app/sw.js scope /app/

The PWA shell MUST register the full Workbox-managed service worker at `/app/sw.js` with `{ scope: '/app/' }`. The SW MUST be generated by `@vite-pwa/astro` configured with `strategies: 'generateSW'`, `registerType: 'prompt'`, `clientsClaim: false`, `skipWaiting: false`, `cleanupOutdatedCaches: true`. The SW MUST own the offline cache for the app's HTML/JS/CSS, font assets, and image assets per the existing `runtimeCaching` rules. The full SW MUST be served with HTTP headers `Cache-Control: no-cache, no-store, must-revalidate` and `Service-Worker-Allowed: /app/`. The full SW MUST NOT intercept any request outside its `/app/` scope.

#### Scenario: Full SW registers with /app/ scope
- **GIVEN** the PWA shell loads at `/app/` in a browser that supports service workers
- **WHEN** the registration completes
- **THEN** `navigator.serviceWorker.getRegistration('/app/')` resolves to a registration whose `active.scriptURL` ends with `/app/sw.js` and whose `scope` is `https://tricho.app/app/`

#### Scenario: Full SW does not affect marketing requests
- **GIVEN** both SWs are registered
- **WHEN** the user navigates from `/app/` back to `/`
- **THEN** the request to `/` is handled by the thin SW's pass-through
- **AND** the full SW's `fetch` handler is not invoked
- **AND** the marketing page renders without any `/app/` cache contamination

#### Scenario: Full SW served with /app/ scope-allowed header
- **GIVEN** the production deploy
- **WHEN** `https://tricho.app/app/sw.js` is fetched
- **THEN** the `Service-Worker-Allowed` response header is `/app/`

### Requirement: Installed PWA launches at /app/

When the user installs the PWA from any surface (marketing landing, app shell, browser-native install prompt), launching the installed icon MUST navigate to `/app/` (the manifest's `start_url`). The launched view MUST be either the welcome wizard (no vault yet), the locked screen (vault present, DEK not in memory), or the unlocked shell (vault unlocked) per the existing `welcome-onboarding-wizard`, `locked-screen`, and AppShell view-routing logic. The marketing landing MUST NOT be reachable as the launched view in standalone mode.

#### Scenario: Fresh install lands on the wizard at /app/
- **GIVEN** a user installed the PWA on a device with no prior `tricho-keystore` rows
- **WHEN** the user taps the home-screen icon
- **THEN** the browser launches with `display-mode: standalone`
- **AND** the URL bar (or equivalent) shows `https://tricho.app/app/` as the start URL
- **AND** the welcome wizard renders

#### Scenario: Returning user lands at /app/ and routes by vault state
- **GIVEN** a user installed the PWA and has a `tricho-keystore` row but the DEK is not in memory (cold start)
- **WHEN** the user taps the home-screen icon
- **THEN** the launched URL is `/app/`
- **AND** AppShell routes to the `'locked'` view per the `locked-screen` capability

### Requirement: SPA fallback for /app/* deep links

Cloudflare Pages MUST serve `dist/app/index.html` (HTTP 200) for any request matching `/app/*` that does not correspond to a real file in `dist/app/`. This is configured via `_redirects` with the line `/app/* /app/index.html 200`. The fallback MUST preserve the original requested path in `window.location.pathname` so the React-side router (or AppShell view-resolution logic) can interpret deep links and reloads.

#### Scenario: Reload on a deep link serves the SPA shell
- **GIVEN** a user is on `/app/clients/abc-123` (a client-side route inside the app)
- **WHEN** the user reloads the page
- **THEN** the response is HTTP 200 with the body of `dist/app/index.html`
- **AND** `window.location.pathname` is `/app/clients/abc-123` after load

#### Scenario: Direct navigation to a deep link works
- **GIVEN** a fresh tab navigates directly to `https://tricho.app/app/settings`
- **WHEN** the response loads
- **THEN** the response is HTTP 200 with the SPA shell HTML
- **AND** the AppShell renders the requested view (or the wizard / locked screen if pre-unlock state requires it)

### Requirement: /app trailing-slash normalization

A request to `/app` (no trailing slash) MUST 301-redirect to `/app/` so that the manifest's `start_url: '/app/'`, internal `<a href="/app/">` links from the marketing site, and the SW scope-matching algorithm all agree on the canonical form. This is configured via `_redirects` with the line `/app /app/ 301`.

#### Scenario: Request to /app redirects to /app/
- **GIVEN** the production deploy
- **WHEN** `https://tricho.app/app` is requested
- **THEN** the response status is 301
- **AND** the `Location` header is `https://tricho.app/app/`

### Requirement: PWA-storage-origin floor moves with the wizard to /app/

The pre-unlock surface's PWA-vs-browser detection (`window.matchMedia('(display-mode: standalone)').matches || window.navigator.standalone === true`) MUST run on the `/app/` route. The "browser-mode hard-stop" requirement (the wizard MUST NOT advance past Step 1 in browser mode) MUST apply at `/app/`. Visiting `/app/` in a regular browser tab MUST render the wizard in browser mode at Step 1; visiting `/app/` from the installed PWA icon (standalone mode) MUST render the wizard at Step 2 (or skip the wizard entirely if a vault exists). The marketing site at `/` MUST NOT host any wizard surface and MUST NOT contribute to PWA storage; clicking through from `/` to `/app/` is a normal in-origin navigation and the wizard's detection runs at the destination.

#### Scenario: Browser tab at /app/ starts wizard at Step 1
- **GIVEN** a regular browser tab opens `https://tricho.app/app/`
- **WHEN** the wizard mounts
- **THEN** the launch-mode detection reports `'browser'`
- **AND** the wizard renders at Step 1 with install instructions

#### Scenario: Standalone launch at /app/ starts wizard at Step 2
- **GIVEN** the installed PWA is launched from the home-screen icon
- **WHEN** the wizard mounts at `/app/`
- **THEN** the launch-mode detection reports `'pwa'`
- **AND** the wizard renders at Step 2

### Requirement: No cross-scope SW interference

The two service workers MUST NOT both attempt to control the same client. A page at `/` MUST be controlled exclusively by the thin SW (or no SW, if the thin SW is still installing). A page at `/app/` or any path under `/app/` MUST be controlled exclusively by the full SW. The system MUST NOT define a SW at any path or scope that overlaps another registered scope.

#### Scenario: Marketing page is controlled by the thin SW
- **GIVEN** the user is on `https://tricho.app/`
- **AND** both SWs have completed activation
- **WHEN** `navigator.serviceWorker.controller` is read
- **THEN** its `scriptURL` ends with `/sw.js` (the thin SW)
- **AND** it does not end with `/app/sw.js`

#### Scenario: App page is controlled by the full SW
- **GIVEN** the user is on `https://tricho.app/app/`
- **AND** both SWs have completed activation
- **WHEN** `navigator.serviceWorker.controller` is read
- **THEN** its `scriptURL` ends with `/app/sw.js`
- **AND** it does not end with `/sw.js` (root)
