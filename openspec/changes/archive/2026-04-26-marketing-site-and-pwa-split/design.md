## Context

The repository today is a single-package Astro 5 PWA at the root: `astro.config.mjs`, `src/`, `public/`, `package.json`, `dist/`, `CNAME`. It builds with `npm run build` and deploys to GitHub Pages on every push to `main` via `.github/workflows/deploy.yml`. The Web App Manifest, generated by `@vite-pwa/astro`, advertises `start_url: '/'` and `scope: '/'`. The service worker output is at `/sw.js` with scope `/`. There is no marketing surface; visiting `https://tricho.app/` mounts the welcome wizard's "install me first" Step 1.

This works for an internal alpha but is wrong for a launch:

- Search engines and AI crawlers see only the SPA loader. There is no indexable copy describing what Tricho is, what it costs, or how data is encrypted. Indexing the wizard would actively hurt — the wizard's copy is install instructions, not a product description.
- Social previews (Twitter, Slack, iMessage) of `tricho.app` show whatever the SPA can render in the bot's first paint, which is nothing.
- The `tricho.app/` URL is wired to do exactly one thing — bootstrap the wizard. There is no place to add an About page, a Pricing page, a Help portal, a blog, legal pages, OG images, or a sitemap without colliding with the wizard's catch-all routing assumptions.
- Cloudflare-class deployment ergonomics (per-PR previews, per-route headers, atomic rollback) are unavailable on GitHub Pages.

The architectural target is described in the user-supplied implementation spec: one Cloudflare Pages project, one origin, marketing at `/`, PWA at `/app/`, two service workers (thin pass-through at `/sw.js` scope `/`, full Workbox at `/app/sw.js` scope `/app/`), independent `web/` and `app/` packages with no monorepo tooling, a `scripts/merge-dist.mjs` orchestrator, GitHub Actions CI/CD with path-filtered tests and PR previews, and an `app-v*` semver release flow. The change preserves all zero-knowledge invariants: the marketing site is plain HTML/CSS that never sees plaintext, never holds a DEK, never opens a `VaultDb`, and never speaks to CouchDB or `tricho-auth`. The IndexedDB origin (`tricho.app`) is unchanged, so any keystore data already persisted survives the cutover — although since this is pre-launch, no production users exist yet.

Stakeholders: the operator (a single founder; needs the deploy pipeline to be observable and the rollback to be one click), the practitioner (single hairdresser/trichologist; will discover Tricho via Google or word-of-mouth, expects a credible marketing surface before they will install anything), and the future content contributor (must be able to write a blog post or a help article in MDX without learning the PWA's React internals).

The architectural constraints from `openspec/project.md` are non-negotiable: server sees ciphertext + OAuth identity only; AAD binds every payload to `{vaultId, docId}`; the Recovery Secret is the offline fallback and never touches the server. This change does not touch any of those — it reorganises the static surface and the build/deploy pipeline only — but every decision below is checked against them anyway.

## Goals / Non-Goals

**Goals:**

1. The apex `tricho.app/` serves a fully static, pre-rendered marketing landing page that any crawler (Googlebot, GPTBot, PerplexityBot, ClaudeBot) can ingest in one round-trip, with every SEO header (canonical, OG, Twitter Card, JSON-LD) on the page at build time.
2. The PWA shell continues to work end-to-end — install, unlock, sync, idle-lock, settings, account deletion — with its routing rebased to `/app/`. No requirement currently keyed on the URL `/` becomes unsatisfiable; the wizard, the locked screen, the unlocked shell, and every existing test continue to pass.
3. The two surfaces are visually indistinguishable as "the same product": same brand wordmark, same paper-grain texture, same Fraunces serif headings, same Patrick Hand body, same copper accents, same theme colors. A user who lands on `/` and clicks through to `/app/` does not perceive a context switch.
4. PWA install criteria are satisfied on the apex via a thin pass-through service worker (`/sw.js`) whose only job is to exist; the full Workbox-managed SW lives at `/app/sw.js` and owns offline cache + asset versioning + push notifications. The two SWs MUST NOT cache each other's surface, and MUST NOT race with each other.
5. Every PR gets a unique, shareable preview URL deployed automatically; reviewers can click through both the marketing site and the app shell on the preview before approving. The URL is posted as a PR comment so it's discoverable without leaving GitHub.
6. The PWA gets a real `vX.Y.Z` semver story: bumping the version, tagging `app-vX.Y.Z`, pushing the tag is one workflow that produces a GitHub Release with auto-generated notes; the in-app Settings → About displays the running version, build time, and commit hash so support questions like "what version are you on?" have a deterministic answer.
7. The user-controlled SW update prompt replaces today's silent auto-update so a new SW cannot take over a session that holds an in-memory DEK and an open `VaultDb`. The user sees "Nová verze připravena — restartovat" and chooses when to apply.
8. Operator runbook exists (`ARCHITECTURE.md`) so a future contributor can understand the topology without re-reading this document.

**Non-Goals:**

1. **Cloudflare Workers, Pages Functions, or any serverless surface.** All dynamic behaviour stays where it is today (in `tricho-auth` and CouchDB).
2. **Subdomain split** (`app.tricho.app` etc.). Explicitly rejected by the implementation spec; the install-criteria reasoning depends on one origin.
3. **Monorepo tooling** (Turborepo, Nx, npm/pnpm workspaces, Lerna). Each top-level package has its own lockfile and `node_modules` and is built standalone.
4. **Backend changes.** No change to `tricho-auth`, no change to CouchDB schemas, no change to the JWT contract, no change to `couch_peruser`, no change to the `vault-state` doc.
5. **Auth changes.** No change to OAuth providers, WebAuthn registration, Recovery Secret derivation, PIN derivation, the envelope crypto, or AAD binding. Those are the zero-knowledge invariants and are explicitly preserved.
6. **E2E infrastructure rewrite.** Existing Playwright tests have their `working-directory` and path filters updated; the test code itself is unchanged except for any URL constants that hardcoded `/` and now must read `/app/`.
7. **Marketing-site i18n.** Czech-only at launch. Astro's i18n routing can be added later without touching the apex topology.
8. **Analytics integration.** Operator picks Cloudflare Web Analytics / Plausible / Umami separately. The Base layout exposes a single `<slot name="head-extra" />` for analytics tags so adding one is a one-file change.
9. **Per-locale Web App Manifest variants.** Already a non-goal in `welcome-onboarding-wizard`; restated here.
10. **Editorial content.** The blog and help portals ship as working skeletons with a single welcome post and a single welcome help article. Filling them is a separate workstream.

## Decisions

### D1. PWA routing rebases to `/app/` via Astro's `base` config; the wizard moves with it; no client-side rewriting

**Decision:** Set `app/astro.config.mjs` to `base: '/app/'` and `site: 'https://tricho.app/app'`. Astro's `base` flows into the build output: every emitted asset path, every `import.meta.env.BASE_URL`, every internal `<a href={...}>` and `<img src={...}>` is automatically prefixed. The wizard mounts from `app/src/pages/index.astro` (which becomes `dist/index.html` inside `app/dist/`, which the merge script places at `dist/app/index.html`, served at `/app/`). The PWA shell's existing client-side state machine (`AppShell.tsx`'s `View` enum: `loading | welcome | locked | unlocked | device-limit | …`) is unchanged — it is a single-page application with one HTML entry point and React-driven view switching.

**Alternatives considered:**

- *Move the wizard to `app/src/pages/index.astro` without setting `base`.* Rejected: emitted asset URLs (`/_astro/<hash>.js`) would still be rooted at `/`, conflicting with the marketing site's identically-named hashed assets and producing 404s after the merge. Astro `base` is the supported, single-config solution.
- *Use a Cloudflare Worker to rewrite `/` requests to the marketing site and `/app/*` requests to the app, both served from the same root URL space.* Rejected: workers are out of scope per the implementation spec; the static-only model is sufficient and the Worker would add a runtime failure mode for no benefit.
- *Put the wizard at `/app/index.html` directly with hand-built relative URLs.* Rejected: error-prone and brittle — every internal navigation would need manual prefixing. Astro `base` does this once, correctly.

**Why this is right:** the rebase is a one-config-line change that the framework was designed for. It preserves the SPA's internal model (one HTML file, React routing) while satisfying the topology constraint (marketing at `/`, app at `/app/`).

**Threat-model delta:** none. The IndexedDB origin remains `tricho.app`; the keystore data persists across the cutover. The same-origin-only fetches (`/auth/*`, `/_users/*`, `/userdb-*/`) are unaffected — they were already same-origin and remain so. The CSP still scopes `connect-src 'self'`.

### D2. Two service workers, two scopes, no overlap; thin SW is hand-written, full SW remains `@vite-pwa/astro`

**Decision:** A 15-line `shared/sw.js` registered at `/sw.js` with scope `/` handles only `install` (`skipWaiting`), `activate` (`clients.claim`), and `fetch` (no-op pass-through, never calls `event.respondWith`). Its purpose is to satisfy the PWA install criteria on the apex (Chrome and Edge require a registered SW *plus* a manifest *plus* HTTPS to surface the install prompt). It MUST NOT intercept any request. The full SW is generated by `@vite-pwa/astro` inside `app/`, with explicit `scope: '/app/'` in the registration call, output filename `sw.js` (so it lives at `app/dist/sw.js` → `dist/app/sw.js` → served at `/app/sw.js`), and `Service-Worker-Allowed: /app/` header from `_headers`.

**Alternatives considered:**

- *One SW at `/sw.js` scope `/` that handles both surfaces.* Rejected on two grounds: (a) the marketing site's caching needs (Cloudflare-edge-cached, browser-cached via `Cache-Control` headers, immutable for hashed assets) are different from the app's (Workbox precache, runtime cache strategies, offline fallback); colocating them in one SW would require Workbox to ignore everything outside `/app/*`, defeating its model. (b) The PWA's `clientsClaim: false` + `skipWaiting: false` semantics for user-controlled updates would also gate the marketing site's SW updates, which is wrong — the marketing site has no DEK and no `VaultDb` to protect.
- *No thin SW; rely on `/app/sw.js` alone for install criteria.* Rejected: the install prompt must fire on the page that links the manifest. If the user lands on `/` (marketing), no SW is registered there, the prompt never fires, and install discoverability collapses.
- *Thin SW also caches the marketing landing for offline.* Rejected: explicitly disallowed by the implementation spec ("no caching, no offline logic"). The marketing site's offline story is handled by Cloudflare's edge cache + browser cache, not a SW. Caching it in a SW would create stale-content risk during launches.

**Why this is right:** two SWs with non-overlapping scopes is the standard browser model (each scope can have at most one controlling SW). The thin SW is small enough to audit at a glance; the full SW is identical to today's except for its scope.

**Threat-model delta:** the thin SW could in principle be tampered with to inject a malicious response. Mitigation: it never calls `event.respondWith`, so the browser handles every request normally; a tampered version would be visible in source view; the file is served with `Cache-Control: no-cache, no-store, must-revalidate` so a poisoned cache cannot pin a malicious copy; integrity is the same as for any static asset on Cloudflare Pages. Threat-model relative to today: equivalent.

### D3. Single shared manifest, linked from both surfaces; install prompt fires from either

**Decision:** `shared/manifest.webmanifest` is the single source of truth. The merge script copies it to `dist/manifest.webmanifest`, served with `Content-Type: application/manifest+json` and `Cache-Control: public, max-age=3600` (short cache so name/icon updates propagate within an hour). Both the marketing landing's `<head>` and the PWA shell's `<head>` link the same path: `<link rel="manifest" href="/manifest.webmanifest">`. The manifest sets `start_url: '/app/'`, `scope: '/app/'`, `display: 'standalone'`. After install, the icon launches `/app/` directly regardless of which surface initiated the install prompt.

**Alternatives considered:**

- *Two manifests — one for marketing, one for the app.* Rejected: a single browser cannot install the same origin twice, so having two manifests is meaningless and only confuses the install criteria heuristics.
- *Manifest linked from `/app/` only.* Rejected: the marketing landing is the discoverability surface; install prompts must fire there too.
- *Manifest linked from `/` only.* Rejected: a user who lands directly on `/app/` (deep link from a tweet, blog post, etc.) in browser mode would see the wizard's "install me" Step 1 with no install affordance, because the install prompt would not fire on a page that doesn't link the manifest.

**Why this is right:** one manifest, two link tags, one install target. The browser de-duplicates and the install criteria are satisfied on both surfaces.

### D4. PWA SW registration switches to `registerType: 'prompt'`; in-app banner gates `SKIP_WAITING`

**Decision:** `@vite-pwa/astro` config inside `app/astro.config.mjs` changes from `registerType: 'autoUpdate'` (today) to `registerType: 'prompt'`. Workbox config gains `clientsClaim: false`, `skipWaiting: false`, `cleanupOutdatedCaches: true`. The PWA registers the SW manually in `app/src/main.ts` with `{ scope: '/app/' }`. A new `app/src/lib/sw-update.ts` exposes a nanostore `swUpdate$ = atom<{ waiting: ServiceWorker | null }>` that the registration handler updates when `registration.waiting` becomes non-null. `AppShell.tsx` renders an `<UpdateBanner>` when `swUpdate$.get().waiting !== null`; tapping the banner posts `{ type: 'SKIP_WAITING' }` to the waiting SW, listens for `controllerchange`, then `window.location.reload()`s.

**Alternatives considered:**

- *Keep `registerType: 'autoUpdate'`.* Rejected for safety: `autoUpdate` flips `clientsClaim: true` and `skipWaiting: true` under the hood, meaning a new SW takes over the page mid-session. The PWA holds an in-memory DEK and an open `VaultDb` while the user is unlocked; an SW takeover during, say, an active sync round can race with PouchDB's leveldown adapter and the encrypted-payload fetches. The user-controlled prompt is one extra click and removes the entire class of mid-session race.
- *Background-sync the new SW silently, prompt only after `idle-lock` fires.* Rejected as over-engineered for a one-developer team; the simple banner is sufficient.

**Why this is right:** the cost is one banner and one tap; the benefit is eliminating an entire class of mid-session SW-takeover race conditions that would be hard to reproduce and harder to debug.

**Threat-model delta:** tightens the contract. Today an attacker who compromised the build pipeline could push a malicious SW that takes over mid-session and exfiltrates plaintext as it's decrypted in memory. With user-gated `SKIP_WAITING`, the attacker must wait for the user to tap the update banner — the user has the chance to read release notes, decide whether to apply, and (crucially) the new SW only activates when there is no in-memory DEK to protect (the reload happens *before* the next unlock). This is a real, if small, defense-in-depth improvement.

### D5. Cloudflare Pages Direct Upload via Wrangler; not the GitHub-Pages-style integration; not Pages Git integration either

**Decision:** Create the Pages project once with `wrangler pages project create tricho --production-branch=main`. This produces a Direct Upload project with no Git integration — Cloudflare does not poll the repo, does not run builds, does not see the source tree. All deploys come from `cloudflare/wrangler-action@v3` invoking `wrangler pages deploy dist --project-name=tricho --branch=<branch> --commit-hash=<sha>`. PR runs use the branch name as the deploy alias (Cloudflare creates `<branch>.tricho.pages.dev`); pushes to `main` deploy to production.

**Alternatives considered:**

- *Cloudflare Pages Git Integration (Cloudflare watches GitHub, runs the build itself).* Rejected: the build must run our `merge-dist.mjs` orchestrator, which Cloudflare's stock build environment doesn't know about. We could configure a custom build command, but then we lose the ability to skip side-builds on path-filtered PRs and we lose direct control over Node version and caching strategy. GitHub Actions is already where our tests run; deploying from there keeps the entire pipeline in one place.
- *Cloudflare Workers + Workers Sites.* Rejected: out of scope per the implementation spec.
- *Stay on GitHub Pages.* Rejected: no per-PR previews, no per-route headers (which we need for `Service-Worker-Allowed`, CSP, and immutable asset cache), no atomic rollback.

**Why this is right:** Direct Upload + GitHub Actions decouples build (where we have full control) from delivery (which is a `wrangler pages deploy` one-liner). The pattern is well-trodden — Cloudflare's own examples assume it for non-trivial projects.

### D6. Path-filtered CI: `dorny/paths-filter@v3` decides which side rebuilds

**Decision:** The first job (`changes`) runs `dorny/paths-filter@v3` with three filters: `web` (`web/**`), `app` (`app/**`), `shared` (`shared/**`), `infra` (`scripts/**`, `_headers`, `_redirects`, `.github/workflows/**`). Subsequent jobs (`test-web`, `test-app`) gate on the relevant filter PLUS `shared` and `infra` (any change to shared assets or build orchestration triggers both sides). The `build-and-deploy` job runs unconditionally (so even a docs-only PR gets a preview URL) but downloads artifacts from the test jobs when they ran and rebuilds locally when they didn't.

**Alternatives considered:**

- *Always rebuild both sides.* Rejected: a copy edit to a blog post would re-run the entire app test suite (~5 min today, growing). Path filtering is the standard pattern for skipping unaffected work.
- *Skip the deploy on PRs that only touch `web/`.* Rejected: the deploy is the *whole point* of the PR — reviewers need to click through the preview. Path filtering applies to *which side rebuilds*, not to whether deploy runs.
- *Use GitHub Actions concurrency groups instead of path filtering.* Rejected: concurrency groups handle "cancel previous runs of the same PR", which we do also want (set `cancel-in-progress: true` for PRs), but they don't address "skip building the side that didn't change".

**Why this is right:** path filtering keeps PR feedback time tight; concurrency groups keep CI cost from blowing out on rapid pushes. They compose cleanly.

### D7. `app-v*` semver; `web-v*` reserved; `app/package.json` `version` is canonical

**Decision:** PWA version lives in `app/package.json` `version`. Bumping is `cd app && npm version --no-git-tag-version <patch|minor|major>`, then a manual commit and a `git tag app-v$(node -p "require('./package.json').version")` to produce a prefix-namespaced tag. The release workflow (`release-app.yml`) triggers on `app-v*` tag push. The marketing site does not need semver today; if it later needs editorial-changelog tags, `web-v*` is reserved by convention. No `v*` (unprefixed) tags are used so the namespace stays clean.

**Alternatives considered:**

- *Use unprefixed `v*` tags for the PWA.* Rejected: forecloses on `web-v*` later and conflates the two products. Prefixing is cheap and self-documenting.
- *Use `npm version` directly so it creates the tag automatically.* Rejected: `npm version` creates an unprefixed `vX.Y.Z` tag by default; getting the prefix requires either configuring `tag-version-prefix` in `app/.npmrc` (ergonomic but a per-package side channel) or running `--no-git-tag-version` and tagging manually. The implementation spec picks the manual-tag form, which is more transparent.

**Why this is right:** explicit is better than implicit when the tags drive a release workflow. The prefix makes filtering trivial (`git tag --list 'app-v*' --sort=-v:refname`) and leaves room for `web-v*`.

### D8. Build constants injected via Astro's Vite `define`; no runtime fetch of version metadata

**Decision:** `app/astro.config.mjs` reads `app/package.json` at config-load time and passes `__APP_VERSION__: JSON.stringify(pkg.version)`, `__APP_BUILD_TIME__: JSON.stringify(new Date().toISOString())`, `__APP_COMMIT__: JSON.stringify(process.env.GITHUB_SHA?.slice(0, 7) || 'dev')` into Vite's `define` block. TypeScript declarations live in `app/src/vite-env.d.ts`. Settings → About reads them directly: `<dd>{__APP_VERSION__} · {__APP_COMMIT__}</dd>`. No runtime HTTP fetch (would be a no-op offline anyway), no fetched manifest version.

**Alternatives considered:**

- *Bake a `version.json` into `dist/app/` and fetch it at runtime.* Rejected: works offline only after the SW caches it; adds network failure modes for no benefit.
- *Use a bare `import pkg from '../package.json'` in the React tree.* Rejected: ships the whole package.json (devDependencies, scripts, name) in the bundle; `define` ships only the strings we want.

**Why this is right:** `define` is the framework-blessed way to inject build-time constants. It's tree-shakable, type-safe (with the `.d.ts`), and offline-correct.

### D9. `_headers` CSP is derived from observed app traffic, not copy-pasted

**Decision:** Before merging the PR, run the existing PWA in production mode (`npm run build && npm run preview`), open the welcome wizard, complete OAuth (mock or real), reach the unlocked schedule view, observe Network panel for every external origin (font CDNs if any, `tricho-auth` API, CouchDB API, OAuth provider redirects, Stripe.js if billing pages render). Whitelist exactly those in `connect-src`, `script-src`, `style-src`, `img-src`, `font-src`, `frame-src`. The implementation spec ships a placeholder CSP under `/app/*`; the placeholder is intentionally tight (`connect-src 'self'`) and must be relaxed only as far as observed traffic requires. Any relaxation is documented in the PR description with the URL that needed it.

**Alternatives considered:**

- *Permissive CSP (`script-src 'unsafe-inline' 'unsafe-eval' *`).* Rejected: defeats the point of having a CSP at all; eliminates a defense-in-depth layer the zero-knowledge contract relies on (an XSS in the wizard could exfiltrate the in-memory DEK).
- *No CSP, rely on `X-Content-Type-Options` and trust the app code.* Rejected on the same grounds.
- *Strict CSP with `'strict-dynamic'` and a build-time nonce per script tag.* Considered, deferred: the existing app uses inline `<script>` blocks for the SW registration and the i18n bootstrap; converting them to nonce-based requires an Astro middleware that doesn't ship with `output: 'static'`. We'll start with the observed-origin allowlist and tighten to `'strict-dynamic'` in a follow-up if the nonce work is justified.

**Why this is right:** the CSP must reflect reality, not aspirations. Observing real traffic is one hour of work and produces a CSP that won't break unexpectedly in production.

**Threat-model delta:** a real CSP with `connect-src` limited to `self` plus the auth/sync hosts means an XSS injection in the app cannot exfiltrate the in-memory DEK to an attacker-controlled URL — the browser blocks the request. This is a meaningful improvement over no CSP today.

### D10. `_redirects` SPA fallback is `/app/* /app/index.html 200`; the `/app` trailing-slash redirect is `/app /app/ 301`

**Decision:** Two lines in `_redirects`. The `200` rewrite for `/app/*` makes deep links (e.g. `tricho.app/app/clients/abc-123`) and reloads-after-deep-link work — Cloudflare serves `dist/app/index.html` and the React router takes over client-side. The `301` redirect normalizes `tricho.app/app` (no trailing slash) to `tricho.app/app/` so internal `<a href="/app/">` links and the `start_url: '/app/'` agree on the canonical form.

**Alternatives considered:**

- *Use `/app/* /app/index.html 200` and skip the `301` redirect.* Risky: the manifest's `start_url: '/app/'` would not match a request to `/app` (no trailing slash) under some browser SW scope-matching algorithms, leaving the installed app in an in-scope-or-out-of-scope grey zone. Cheap to redirect; cheap to be explicit.
- *Use a Cloudflare Worker for routing.* Out of scope.

**Why this is right:** two lines, no runtime cost, fully declarative, and the same file format Cloudflare Pages, Netlify, and Vercel all read.

### D11. `merge-dist.mjs` is the single integration point; it has no external dependencies; it validates collisions and exits non-zero on failure

**Decision:** The script uses Node 22+ built-ins (`node:fs/promises`, `node:path`, `node:url`) only. Algorithm:

1. `await fs.rm('dist', { recursive: true, force: true })` then `await fs.mkdir('dist', { recursive: true })`.
2. `await fs.cp('web/dist', 'dist', { recursive: true })`.
3. Before copying app: `await fs.cp('app/dist', 'dist/app', { recursive: true })`. Detect collisions by reading every file in `web/dist` first (recursive walk) and asserting none match a file we're about to write into `dist/app/<X>` where the relative path *also* matches an entry from web. (For example, both sides emitting `/sw.js` would collide; the merge script must reject this and tell the operator which file collided.)
4. `await fs.copyFile('shared/manifest.webmanifest', 'dist/manifest.webmanifest')`.
5. `await fs.copyFile('shared/sw.js', 'dist/sw.js')`.
6. `await fs.cp('shared/icons', 'dist/icons', { recursive: true })`.
7. `await fs.copyFile('_headers', 'dist/_headers')` and `await fs.copyFile('_redirects', 'dist/_redirects')`.
8. Print a summary (file count + size) to stdout for the CI log.

`validate-build.mjs` runs after `merge-dist.mjs` and asserts: `dist/index.html` exists and references `/manifest.webmanifest`; `dist/app/index.html` exists; `dist/sw.js` exists and matches the byte length of `shared/sw.js`; `dist/app/sw.js` exists; `dist/_headers` and `dist/_redirects` exist; the marketing `<head>` contains a `<link rel="manifest" ...>` and a `<script>` block registering `/sw.js`.

**Alternatives considered:**

- *Use `cpx` or `fs-extra`.* Rejected: external dependency for what Node 22 has built-in.
- *Use a `Makefile`.* Rejected: cross-platform Make is a known pain; Node script is ergonomic on macOS, Linux, and CI.
- *Inline the merge logic into the GitHub Actions YAML.* Rejected: untestable, unrunnable locally, and locks us into Actions if we ever migrate CI.

**Why this is right:** one file, no deps, runs anywhere Node 22 runs, fails loud. The validation script catches the worst failure modes (missing manifest link, missing SW, collision-induced overwrite) before deploy.

### D12. The thin SW lives in `shared/`; the full SW lives in `app/`; both are version-controlled

**Decision:** `shared/sw.js` is the actual file the merge script copies to `dist/sw.js`. `app/dist/sw.js` is generated by `@vite-pwa/astro`'s Workbox build; it is never committed (it's gitignored along with `app/dist/`). The merge script copies `app/dist/sw.js` to `dist/app/sw.js` as part of the recursive `app/dist/` → `dist/app/` copy. There is no race or collision because they live at different paths.

**Alternatives considered:**

- *Symlink the thin SW from both `shared/` and `web/public/`.* Rejected: cross-package symlinks are fragile in CI checkouts on Windows agents (we don't use Windows agents but the code travels), and the merge script provides a clearer single-direction copy.

### D13. `web/` does not import from `app/` or vice versa; brand tokens are duplicated across packages

**Decision:** The brand tokens (CSS custom properties for `--copper`, `--ink-espresso`, `--surface`, the paper grain texture URL, the Fraunces / Patrick Hand / Geist font face declarations) are duplicated into `web/src/styles/tokens.css` from `app/src/styles/tokens.css` (or wherever they live in the app today). A small comment at the top of each file points to the other for parity audits. There is no monorepo tooling and no cross-package import.

**Alternatives considered:**

- *Put tokens in `shared/styles/tokens.css` and have both packages copy at build time.* Considered. Rejected because it creates a third source of truth that both packages must remember to sync from. Duplicating once is cheaper than maintaining a copy script + watching for drift.
- *Adopt npm workspaces just to share tokens.* Rejected by the implementation spec; not worth the tooling delta for a 200-line CSS file that changes ~once a year.

**Why this is right:** the tokens evolve slowly and the duplication is small. If the tokens were a 5000-line design system with hundreds of components, this calculus changes — but they are not, today.

### D14. The marketing-site `<head>` uses Astro layout slots; per-page metadata is data, not boilerplate

**Decision:** `web/src/layouts/Base.astro` accepts a `Props` interface (`title`, `description`, `canonical?`, `ogImage?`, `ogType?`, `noIndex?`) and renders the entire `<head>`. Every page passes these as props; no page hand-writes a `<meta>` tag. Blog and help posts derive their props from MDX frontmatter (Astro Content Collections). The sitemap is generated by `@astrojs/sitemap` from the same source. JSON-LD is rendered conditionally based on page type (`Organization` on `/`, `SoftwareApplication` on `/` + `/pricing`, `Article` on blog posts).

**Alternatives considered:**

- *Hand-write `<head>` per page.* Rejected: drift, omissions, and copy-paste bugs are inevitable.
- *Use a non-Astro framework (Next.js static export).* Rejected per the implementation spec, which lists Astro as the preferred default. Next.js static export is acceptable per the spec but not preferred; Astro's content collections + MDX + `@astrojs/sitemap` map directly onto our needs.

**Why this is right:** the layout owns the contract; pages own the content. Adding a new page is one MDX file or one `.astro` file, and SEO is correct by construction.

### D15. iOS install banner is rendered as a small Astro component, hidden by default, revealed by a tiny inline script

**Decision:** `web/src/components/InstallBanner.astro` renders a hidden `<div data-install-banner hidden>` with the iOS instructions. An inline `<script>` at the end of `Base.astro` runs `if (/iPhone|iPad|iPod/.test(navigator.userAgent) && !navigator.standalone) { document.querySelector('[data-install-banner]')?.removeAttribute('hidden'); }`. No client-side framework hydration; no Astro island; ~10 lines of JS. The banner is dismissable (click sets `localStorage.setItem('install-banner-dismissed', '1')`); the inline script also checks that key and exits if set.

**Alternatives considered:**

- *Render the banner as an Astro React island.* Rejected: ships React for ~10 lines of conditional reveal logic. Unjustifiable for a marketing landing's performance budget.
- *Render unconditionally and hide via `@media`.* Rejected: there is no media query for "iOS Safari and not standalone".
- *Use the `beforeinstallprompt` event.* Rejected for iOS: iOS Safari does not fire `beforeinstallprompt`. The whole reason this banner exists is *because* iOS doesn't fire that event.

**Why this is right:** ten lines of JS for a JS-feature-detect; no framework cost; degrades gracefully (the banner just doesn't show on browsers without JS, which is correct because those browsers can't install a PWA either).

### D16. The wizard's existing scenarios that key on "the root route" are reworded to "the `/app/` route"

**Decision:** The `welcome-onboarding-wizard` MODIFIED block in this change rewords two requirements:

- **Requirement: Single welcome route owns all pre-unlock UI** — language updates to reference `app/src/pages/index.astro` and the `/app/` route. The existing scenarios "Visiting the root..." become "Visiting `/app/`...". The substantive behaviour (single `<WelcomeScreen>`, no legacy auth screens, single `welcome` view enum) is unchanged.
- **Requirement: Launch mode is detected on every mount and never persisted** — the scenario "Re-opening in browser after uninstall" updates the URL referenced from `tricho.app/` to `tricho.app/app` (the canonical browser-mode entry within the new app scope). The `display-mode: standalone` detection is unchanged.

All other requirements in `welcome-onboarding-wizard` are URL-agnostic (they reference the wizard's internal state machine, not the URL it's mounted at) and are not modified.

**Alternatives considered:**

- *Mark the entire `welcome-onboarding-wizard` capability as MODIFIED and copy every requirement.* Rejected: needlessly noisy. OpenSpec deltas should reflect what changes.
- *Add a NEW capability `pwa-shell-routing` and reference it from `welcome-onboarding-wizard` but don't modify the wizard spec at all.* Rejected: the wizard spec literally says "mounted from `src/pages/index.astro` ... whenever the root route is loaded". Leaving that text unchanged after the rebase would let the spec lie.

**Why this is right:** OpenSpec MODIFIED deltas exist for exactly this case. Two surgical edits keep the spec true without bloating the change.

### D17. Cutover strategy: hard switch on merge to main; CNAME removed; GitHub Pages deactivated

**Decision:** This is pre-launch. There are no production users to migrate. The cutover is:

1. Merge the PR to `main`.
2. The new `ci.yml` workflow runs and deploys to Cloudflare Pages (the project must already exist — operator-side prerequisite).
3. Cloudflare Pages serves `tricho.pages.dev` with the new topology immediately.
4. Operator attaches the custom domain `tricho.app` in the Cloudflare dashboard. DNS for `tricho.app` is updated to point at Cloudflare (CNAME → `<project>.pages.dev` or proxied through Cloudflare DNS).
5. GitHub Pages is disabled in repo settings to prevent the old `deploy.yml` from re-deploying the old surface (it can't anyway because the PR removes the workflow file, but defense in depth).

**Rollback strategy:** Cloudflare Pages keeps every deployment indefinitely (unless the operator deletes them). One click in the Cloudflare dashboard rolls back to the previous deployment. If the rollback target is a pre-cutover deployment (i.e. the entire change is reverted), the marketing surface disappears and the PWA is back at the apex via its old `/sw.js` scope `/`. Since there are no production users, this is functionally a no-op.

**Alternatives considered:**

- *Soft cutover: leave `/` serving the old PWA, only add `/app/` as a new entry, run both for a month, then deprecate `/`.* Rejected: there is nothing to deprecate gracefully (no users), and running both means the manifest situation is muddled (which `start_url` wins?).
- *Stage to `staging.tricho.app` first.* Pre-launch rendering of the marketing site is exactly what PR previews already provide. A separate `staging` subdomain would add operator overhead for no benefit.

**Why this is right:** pre-launch + Cloudflare's instant rollback = the cheapest cutover plan that exists. The complexity is the operator-side one-time setup (project create, custom domain attach, secret config), which is documented in `README.md`.

## Risks / Trade-offs

- **Risk:** `Astro base: '/app/'` breaks an asset path or a hardcoded URL in the existing app (any place that today writes `'/'` or `/foo` instead of `import.meta.env.BASE_URL` or relative URL).
  **Mitigation:** Pre-merge audit. `grep -rn "from '/" app/src/`, `grep -rn "href=\"/" app/src/`, `grep -rn "src=\"/" app/src/` and review every hit. The build itself catches most cases (404s in the preview deploy). The PR's smoke-test acceptance step ("open the preview, exercise install + unlock + sync + idle-lock") catches the runtime-only cases.
- **Risk:** Two SWs at non-overlapping scopes is correct in spec but browsers occasionally surface bugs around scope-matching (especially older Safari).
  **Mitigation:** Manual testing on the PR's preview URL across Chrome (desktop + Android), Edge (desktop), Firefox (desktop), Safari (macOS + iOS) before merge. The thin SW is small enough to remove if a browser misbehaves; the install criteria fall back to "no install prompt" rather than breaking the page.
- **Risk:** Cloudflare's free Pages plan has limits (500 builds/month, 1 build at a time, 20 000 files per deployment). Our build is small but the PR-preview-per-PR pattern can use builds quickly on a noisy week.
  **Mitigation:** `concurrency: { group: ${{ github.workflow }}-${{ github.ref }}, cancel-in-progress: true }` for PR runs cancels in-flight builds when a new commit lands. We are far under the file-per-deploy ceiling. If we ever exceed the build quota, upgrading to the paid tier is $5/month.
- **Risk:** The merge script's collision detection is a shallow check (filename match in the relative tree), not a deep content compare. Two unrelated builds emitting identical paths would still be a bug, but with deep content it might not actually corrupt deployment.
  **Mitigation:** Acceptable. Filename collision is the failure mode that produces broken deploys; deep content match between unrelated builds is overwhelmingly unlikely (file hashes are content-addressed). If we ever hit it, we add the deeper check.
- **Risk:** Switching from `autoUpdate` to `prompt` changes user-facing UX. Returning users will see the "Nová verze připravena" banner the first time they open the new build — they may not know what it means.
  **Mitigation:** Pre-launch, no users exist. After launch, the banner copy is explicit ("Nová verze připravena — restartovat") and the action is one tap. Documented in the help portal.
- **Risk:** Marketing-site i18n is deferred. Czech-only at launch means the en-language SEO surface is missing. Google may rank `tricho.app` lower in non-Czech locales than it otherwise could.
  **Mitigation:** Acknowledged. The marketing-site i18n workstream is noted in `Non-Goals` and can ship as a follow-up using Astro's i18n routing.
- **Risk:** CSP is derived from observed traffic, which means an unexercised code path (e.g. a settings screen that fetches from an origin we didn't observe) could break in production.
  **Mitigation:** The CSP audit happens before merge; the operator opens every screen reachable from the navigation. If a CSP violation does ship, the browser console reports it (and Cloudflare has request logging). The fix is one-line in `_headers`.
- **Trade-off:** Two service workers means two registrations to debug, two cache stories, two `Service-Worker-Allowed` headers. We pay this complexity to get install criteria on the apex while keeping the marketing site cache-friendly. Worth it.
- **Trade-off:** Brand tokens duplicated across `web/` and `app/` will drift if not maintained. Acceptable at current size; revisit if the design system grows.
- **Trade-off:** The install prompt fires on the marketing landing now — the user can install Tricho without ever seeing the wizard's Step 1 install instructions. The wizard's Step 1 still exists for users who landed directly on `/app/` in browser mode, but the main install path now bypasses it. This is the intended behaviour per the implementation spec but represents a UX change from today's "install via the wizard" model. Documented in `welcome-onboarding-wizard` MODIFIED notes; copy of the wizard's Step 1 unchanged.

## Migration Plan

1. **Pre-merge (operator)**:
   - Confirm pre-launch status (no production users).
   - Create `CLOUDFLARE_API_TOKEN` (Cloudflare dashboard → My Profile → API Tokens → Create Token → Custom token, scope: Cloudflare Pages — Edit, account: target account only) and add as repo secret `CLOUDFLARE_API_TOKEN`.
   - Find `CLOUDFLARE_ACCOUNT_ID` (Cloudflare dashboard sidebar) and add as repo secret `CLOUDFLARE_ACCOUNT_ID`.
   - Locally: `npx wrangler login`, then `npx wrangler pages project create tricho --production-branch=main`. This is one-time; subsequent deploys use the secrets.
2. **Implement the change** per `tasks.md`.
3. **PR review**: open the preview URL, exercise `/`, `/about`, `/pricing`, `/blog`, `/help`, then `/app/`, complete a full install + unlock + sync round on Chrome (desktop and Android), Edge, Firefox, Safari (macOS), Safari (iOS). Verify Lighthouse scores on `/` (target: Perf ≥ 95, Acc ≥ 95, BP = 100, SEO = 100, PWA installable). Verify Chrome DevTools → Application → Manifest reports "Installable" with no errors. Verify the iOS install banner appears on iPhone Safari and is hidden on Android Chrome / desktop.
4. **Merge to `main`**: production deploy runs.
5. **Post-merge (operator)**:
   - Cloudflare dashboard → Workers & Pages → tricho → Custom domains → add `tricho.app` (DNS update may be required if not yet on Cloudflare DNS) and `www.tricho.app` (with 301 redirect to apex).
   - GitHub repo → Settings → Pages → set source to "None" (deactivate GitHub Pages).
   - GitHub repo → Settings → Branches → add branch protection rule for `main`: require `Test web`, `Test app`, `Build merged dist & deploy` to pass; require linear history; restrict direct pushes.
   - Optional: enable Cloudflare Web Analytics in the Cloudflare dashboard (one click) — separate workstream, not gated on this change.
6. **Post-launch monitoring**: watch for CSP violation reports in browser console (we don't have a CSP report-uri yet — that's a follow-up). Watch Lighthouse scores monthly via PageSpeed Insights.

**Rollback** (if any of the above goes wrong):

1. Cloudflare dashboard → Workers & Pages → tricho → Deployments → "Rollback to this deployment" on the most recent pre-merge deployment. Site is restored in seconds.
2. If the rollback target is the *initial* deployment (i.e. before any of this change shipped), the previous GitHub Pages-deployed site is no longer reachable (CNAME removed), so re-enable GitHub Pages in repo settings and re-add the `CNAME` file in a hotfix PR. This restores the old `tricho.app` PWA-only surface while we figure out what went wrong.
3. The change does not modify any backend or any user data, so rollback is purely a static-deploy operation. No database migration to undo, no auth change to revert.
