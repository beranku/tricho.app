# Architecture

Current-state snapshot of how Tricho.app is put together. Behavioural
requirements live in `openspec/specs/<capability>/spec.md`; this file
points at them rather than restating SHALL/MUST language.

## Tenets

- **Offline-first PWA.** Everything works without network; sync is opt-in.
- **End-to-end encryption.** The server only ever sees ciphertext.
- **Recovery Secret (RS)** is the durable, server-state-free fallback that
  reconstructs vault access. Lost RS = lost data ŌĆö by design.
- **WebAuthn + PRF** is the day-to-day unlock when the authenticator
  supports it; PIN and RS are explicit fallbacks.

## Two service workers, two scopes, no overlap

- **Thin SW** at `/sw.js` (scope `/`): ~15 lines, three handlers, **no
  caching**. Sole purpose: satisfy install criteria so Chrome/Edge surface
  the install prompt on the marketing landing.
- **Full SW** at `/app/sw.js` (scope `/app/`): generated by
  `@vite-pwa/astro` with `registerType: 'prompt'`, `clientsClaim: false`,
  `skipWaiting: false`. Owns offline cache, asset versioning, and the
  user-controlled update prompt (the `<UpdateBanner>` in the unlocked
  shell).

Updates are **never silent**. When a new SW finishes installing it sits
waiting; the user sees "Nov├Ī verze p┼Öipravena ŌĆö restartovat" and chooses
when to reload. This avoids racing with the in-memory DEK.

## Two packages, one merge

Each top-level package builds independently. There is **no monorepo
tooling** (Turborepo, Nx, npm/pnpm workspaces, Lerna).

```
web/  npm install && npm run build  ŌåÆ  web/dist/   (static marketing)
app/  npm install && npm run build  ŌåÆ  app/dist/   (PWA, base: '/app/')
```

`scripts/merge-dist.mjs` then combines them:

```
dist/                     ŌåÉ web/dist/**
dist/app/                 ŌåÉ app/dist/**
dist/manifest.webmanifest ŌåÉ shared/manifest.webmanifest
dist/sw.js                ŌåÉ shared/sw.js
dist/icons/, dist/og/     ŌåÉ shared/icons/**, shared/og/**
dist/_headers, dist/_redirects ŌåÉ repo root
```

`scripts/validate-build.mjs` then asserts: marketing index links the
manifest and registers `/sw.js`; both SWs exist at the right paths;
manifest's `start_url` and `scope` are `/app/`; sitemap, redirects,
headers all present. Exits non-zero on any failure.

## Document shape

Everything in the per-user database is of shape:

```ts
{
  _id: string,          // "<type>:<uuid>"
  _rev: string,         // PouchDB MVCC revision
  type: 'customer' | 'visit' | 'appointment' | 'photo-meta' | 'vault-state',
  updatedAt: number,    // ms since epoch, used for conflict resolution
  deleted: boolean,     // soft-delete flag
  payload: {            // opaque AES-256-GCM ciphertext
    ct: string,         // base64url
    iv: string,         // base64url
    kid: string,        // vault id (key identifier)
    v: 1,               // schema version
    aad: string,        // context used as additional-authenticated-data
  }
}
```

The only doc that deviates is `vault-state` ŌĆö it holds the already-wrapped
DEK directly, because the wrap is itself opaque ciphertext.

See `openspec/specs/payload-encryption/`, `envelope-crypto/`,
`vault-keystore/`, `vault-state-sync/`.

## Zero-knowledge properties

- `payload` fields are AES-GCM ciphertext; the server holds no key
  material.
- AAD binds every ciphertext to `{vaultId, docId}` ŌĆö a ciphertext cannot
  be spliced from one doc into another.
- The session cookie is the only credential the browser carries across
  the wire; it grants access only to the user's own private `userdb-*`.
- `wrappedDekRs` rides inside a `vault-state` doc because the RS itself
  never touches the server ŌĆö the wrap is useless without it.

## Unlock flow

1. Browser loads `/app/` ŌåÆ `AppShell` hydrates, lists vault states from
   the KeyStore IndexedDB.
2. If a vault exists, the user unlocks via passkey (PRF), PIN, or by
   entering the Recovery Secret.
3. **PRF path:** `getPrfOutput(credentialId, vaultId)` ŌåÆ HKDF-SHA256 with
   the vault's `deviceSalt` ŌåÆ KEK ŌåÆ unwrap `wrappedDekPrf` ŌåÆ DEK in
   memory.
4. **RS path:** user enters Base32 RS ŌåÆ HKDF-SHA256 with `deviceSalt` ŌåÆ
   KEK ŌåÆ unwrap `wrappedDekRs` ŌåÆ DEK in memory.
5. AppShell imports the DEK as a `CryptoKey`, opens the per-vault
   PouchDB via `openVaultDb(vaultId, dekKey)`, and passes the handle to
   the schedule/client/camera components.

See `openspec/specs/passkey-prf-unlock/`, `local-pin-fallback/`,
`recovery-secret/`, `locked-screen/`, `idle-lock/`.

## Sync flow

1. At vault creation, AppShell provisions a CouchDB account via
   `tricho-auth` and JWT-bearer-binds it to the OAuth identity.
2. PouchDB replication uses a `fetch` override that injects the JWT;
   `db.sync(remote, { live: true, retry: true })` runs continuously.
3. The CouchDB JWT proxy validates JWT + subscription + `paidUntil`
   before forwarding each request. On 401 ŌåÆ refresh JWT and retry once
   transparently. On 402 `plan_expired` ŌåÆ throw a typed error and route
   to the Plan screen, **no retry**.
4. Conflicts: docs with `_conflicts` are resolved deterministically by
   picking the highest `updatedAt` revision; losers are soft-deleted.
   Ciphertext is opaque so we can't merge semantically.
5. Photos ride the same replication ŌĆö the cipher blob is an attachment
   on the photo-meta doc.

`appointment.startAt` lives only inside the encrypted `payload`. Schedule
queries scan all `appointment` docs by type via the `[type, updatedAt]`
index, decrypt each row, then filter by `startAt` client-side. Trading a
range query for `O(N_appointments)` decrypts is acceptable for a
single-user practice and preserves zero-knowledge.

See `openspec/specs/live-sync/`, `device-management/`, `billing-plans/`.

bootstrap script in `Layout.astro` reads it before paint to avoid a
lightŌåÆdark flash.

## CI/CD

- `.github/workflows/ci.yml` ŌĆö `dorny/paths-filter@v3` detects which side
  changed; `test-web` and `test-app` run independently and in parallel;
  `build-and-deploy` runs `merge-dist` + `validate-build` and deploys via
  `cloudflare/wrangler-action@v3 pages deploy dist --project-name=tricho`.
  The deploy `--branch` is set from `github.ref_name`, so a push to `dev`
  deploys to `dev.tricho.app` and a push to `main` deploys to
  `tricho.app`.
- `.github/workflows/promote-dev-to-main.yml` ŌĆö fast-forwards `main` to
  `dev`'s tip after four preflight gates (linear history, ahead-of-main,
  no merge commits, green staging CI), tags `prod-YYYY-MM-DD-<shortsha>`,
  and dispatches `ci.yml` on `main`.
- `.github/workflows/release-app.yml` ŌĆö triggers on `app-v*` tags and
  publishes a GitHub Release.

Cloudflare Pages keeps every deployment; rollback is one click in the
dashboard.

See `docs/build-and-deploy.md` for the operator runbook and
`openspec/specs/cloudflare-pages-deploy/`, `release-promotion/`,
`app-release-versioning/`.
